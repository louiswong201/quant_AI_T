# 框架对分钟频的支持评估

本文档评估当前 **Quant Framework** 在**分钟频**（minute-level bars）下的支持程度：哪些能直接满足、哪些需要小改、哪些不满足。

---

## 一、结论概览

| 维度 | 能否满足分钟频 | 说明 |
|------|----------------|------|
| **回测引擎** | ✅ 能 | 按「每一行 = 一个 bar」推进，不假设 bar 是日；分钟数据可跑 |
| **数据层（存储/列）** | ✅ 能 | date 支持 datetime（含时分秒）；Binary 用 int64 纳秒，分钟无问题 |
| **数据层（日期区间过滤）** | ⚠️ 需注意 | 仅传「日期」时 end_date 为当日 0 点，会漏掉当日分钟；需约定或小改 |
| **绩效分析（年化/夏普等）** | ❌ 不能 | 硬编码 252 为「一年 bar 数」，分钟频下年化与波动率会错 |
| **策略与 RAG** | ✅ 能 | on_bar 与 as_of_date 与 bar 粒度无关；分钟 bar 即分钟级时间一致 |
| **指标计算** | ✅ 能 | 按 bar 数滚动（如 ma5=5 根 bar），分钟下即 5 分钟均线 |

**总结**：  
- **能跑**：用分钟 K 线数据可以完成「按分钟逐 bar 回测」，引擎、策略、RAG、指标都兼容。  
- **不能直接满足**：**绩效分析**里的年化收益、年化波动率、夏普、卡玛等是按「252 个交易日」算的，用于分钟数据会**严重失真**，必须按频率扩展（见下文改进建议）。  
- **需注意**：加载分钟数据时日期区间若只传 `end_date="YYYY-MM-DD"`，当前实现会按当日 0 点截断，**最后一天的分钟会漏掉**，需约定 end 为 end-of-day 或改适配器。

---

## 二、逐模块说明

### 2.1 回测引擎（BacktestEngine）

- 逻辑是：取 `data["date"]` 的**有序时间序列**，按 `for i in range(n)` 逐 bar 推进，每 bar 调用 `on_bar(hist, current_date)`，用当前 bar 的 OHLC/close 成交。  
- **没有**「bar = 日」的假设：只要 `date` 列是时间类型且有序，日线、分钟线、tick 聚合 bar 都可以。  
- 变量名 `daily_returns_arr`、`current_date` 在分钟下会显得误导，但不影响正确性；可后续改为 `bar_returns`、`current_time` 等以支持多频率。

**结论**：✅ 能满足分钟频回测。

---

### 2.2 数据层

- **FileAdapter / Parquet / Arrow**：`date` 列用 `pd.to_datetime(...)` 解析；若数据里是带时间的字符串或 datetime（如 `2024-01-02 09:31:00`），会得到分钟级时间戳，**存储与加载都支持分钟**。  
- **Binary 存储**：`date` 存为 int64 纳秒时间戳，**精度足够**分钟（乃至秒/毫秒）。  
- **日期区间**：当前类似 `(df["date"] >= pd.Timestamp(start_date)) & (df["date"] <= pd.Timestamp(end_date))`。若 `start_date`/`end_date` 为 `"YYYY-MM-DD"`，则 `pd.Timestamp(end_date)` 为当日 **00:00:00**，因此「小于等于」会**排除该日 00:00:00 之后的所有分钟**，即**最后一天的分钟 bar 会漏掉**。  
  - **建议**：要么约定「分钟数据时 end_date 传 end-of-day」（如 `"2024-01-31 23:59:59"` 或由调用方转为当日最后一根 bar 的时间），要么在适配器里对「仅日期」的 end_date 自动设为当日 23:59:59（或按数据实际最大时间取）。

**结论**：✅ 存储与列支持分钟；⚠️ 日期区间需注意 end 为 0 点的问题，建议约定或小改。

---

### 2.3 绩效分析（PerformanceAnalyzer）

- 当前实现（节选）：
  - `trading_days = len(portfolio_values)`
  - `years = trading_days / 252`   // **假设 252 bar = 1 年**
  - `annual_return = (1 + total_return) ** (1 / years) - 1`
  - `volatility = np.std(returns) * np.sqrt(252)`   // **年化波动率按 252 日**
- 若传入的是**分钟 bar** 的 `portfolio_values`（例如 1 年 约 6 万根 bar），则：
  - `years = 60000 / 252` 会变成一个巨大的数，**年化收益会被压成接近 0**；
  - 波动率乘 `sqrt(252)` 也会错，应乘的是「每分钟 bar 数对应的年化因子」（例如约 `sqrt(252*240)` 若按 240 分钟/日）。

**结论**：❌ 当前**不能**正确支持分钟频的年化指标；必须引入「频率」或「每年 bar 数」参数后再做年化。

---

### 2.4 策略与 RAG

- **on_bar(data, current_date)**：`current_date` 为当前 bar 的时间戳；分钟数据下即为分钟时间，策略按「当前分钟」决策，语义正确。  
- **get_rag_context(..., as_of_date=current_date)**：回测时传当前 bar 时间，RAG 只检索该时刻之前的文档，**分钟频下时间一致**，无额外问题。

**结论**：✅ 能满足分钟频。

---

### 2.5 指标（indicators）

- 如 MA、RSI、MACD 等是按「bar 数」做滚动（如 `rolling(5)` = 5 根 bar）；在分钟数据下即为 5 分钟均线等，**与频率无关**。

**结论**：✅ 能满足分钟频。

---

## 三、若要「完整满足」分钟频，建议改动

1. **绩效分析**  
   - 增加参数：如 `bars_per_year: Optional[float] = None`，或 `frequency: Literal["daily", "minute"] = "daily"`。  
   - 若为分钟：可设 `bars_per_year = 252 * 240`（或由调用方传入实际分钟数）；`years = len(portfolio_values) / bars_per_year`；年化波动率用 `np.sqrt(bars_per_year)`。  
   - 这样日频仍用 252，分钟频用正确年化因子。

2. **日期区间（适配器）**  
   - 方案 A：文档约定「分钟数据请将 end_date 设为当日最后一根 bar 的时间或 23:59:59」。  
   - 方案 B：在 FileAdapter（及统一入口）中，若发现 `date` 列为带时间的 datetime，且 `end_date` 仅含日期，则自动将 end 设为该日 23:59:59 或按数据 max(date) 截断，避免漏掉当日分钟。

3. **命名与文档**  
   - 在文档中明确：**bar 不限于日**，可以是分钟；`daily_returns` 在非日频下实为「每 bar 收益」；绩效分析在非日频下需传 `bars_per_year` 或 `frequency`。  
   - 可选：引擎/结果里增加 `bar_frequency` 或 `bars_per_year` 字段，供分析器自动识别。

---

## 四、小结表

| 能力 | 日频 | 分钟频（当前） | 分钟频（改完建议后） |
|------|------|----------------|----------------------|
| 按 bar 回测 | ✅ | ✅ | ✅ |
| 数据加载与存储 | ✅ | ✅（区间需注意 end） | ✅ |
| 年化收益/波动率/夏普等 | ✅ | ❌ 错 | ✅ |
| RAG as_of_date | ✅ | ✅ | ✅ |
| 指标（MA/RSI 等） | ✅ | ✅ | ✅ |

**直接回答「能否满足分钟频」**：  
- **能满足**：分钟级**回测运行**、数据读写、策略与 RAG 时间一致、指标计算。  
- **不能满足**：在**不改代码**的前提下，**绩效分析**中的年化与夏普等指标在分钟数据下是**错误**的；需按上文增加频率/`bars_per_year` 后再用分钟数据做绩效分析。
