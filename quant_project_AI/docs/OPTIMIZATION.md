# RAG 与量化框架优化说明

本文档合并了 RAG 模块的**易懂版优化说明**、**循环与低效写法优化**、**极限优化与 Rust 边界**，以及**配置与可观测性**，便于统一查阅。

---

## 一、阅读前：几个词先搞清楚

| 词 | 通俗解释 |
|----|----------|
| **RAG** | 用“检索到的文档片段”增强回答：先查库、再根据查到的内容生成答案。 |
| **嵌入 (embedding)** | 把一段文字变成一列数字（向量），语义相近的文本会得到相近的数字，方便用数学方式做“相似度”检索。 |
| **分块 (chunk)** | 把长文档切成一段段短文本再存，检索时按“段”返回，避免整篇太长。 |
| **实时** | 数据一进来就尽快能查得到；写数据的接口要**马上返回**，不能卡住。 |
| **COW** | Copy-on-Write：写的时候先在新副本上改，改完再一次性替换“当前版本”，读的人永远看到的是完整的一版。 |
| **临界区** | 多线程里“加锁的那一小段代码”；临界区越短，锁占用的时间越少。 |

---

## 二、我们想解决的三类问题

1. **和量化框架脱节**：策略在做买卖决策时，拿不到“新闻/研报”这类非结构化数据，或者拿到的没有和回测日期对齐，容易用到“未来信息”。  
2. **不够实时**：文档一写入就卡在“分块+嵌入”上，主线程被拖慢；检索时又可能被写入挡住，延迟不稳定。  
3. **性能和资源**：向量用列表一条条存、每次检索都临时组大数组，浪费内存和 CPU；嵌入按“单篇文档”一次次调，批处理能力用不上。

下面按**问题 → 我们怎么做 → 带来的好处**来说。

---

## 三、和量化框架怎么“接在一起”

- **做法**：在数据层增加 `RagContextProvider`，与 `DataManager` 平级。策略在构造时传入 `rag_provider`，在 `on_bar` 里调用 `get_rag_context(query, symbol, as_of_date=current_date)`。回测引擎按日期驱动 `on_bar`；时间一致性由“策略总是传 `current_date`”和后续检索的 `as_of_date` 保证。  
- **好处**：回测可复现、不偷看未来；数据源（行情 + RAG）统一从数据层出来。

---

## 四、“实时”我们是怎么做的

**写入路径：队列 + 后台工人**

- `ingest_put(doc)` 只把文档放进有界队列后**马上返回**。队列满了返回失败，不阻塞调用方。  
- 后台 daemon 线程按批取文档：规范化 → 分块 → **一次**批量嵌入 → 写入向量库和关键词索引。  
- 主线程只负责入队，重活都在后台，延迟稳定。

**读取路径：读“快照”，不跟写抢锁**

- **COW（写时复制）**：写在“工作区”追加，一批处理完后复制成新快照并设为“当前可读版本”。读只读这份快照（复制引用 + size），在**锁外**做点积、排序、取 top_k。  
- 读的临界区非常短；写只在更新快照时短暂持锁，检索延迟更稳定。

## 五、计算和内存怎么省、怎么快

### 5.1 向量怎么存、怎么查

- **一块连续内存**：所有向量放在一个 `numpy` 二维数组（`float32`），预分配容量，不够再按比例扩容。  
- **写入时归一化**：在 `add()` 时就把每一行做成 L2 归一化；检索时**只算一次点积**就等于余弦相似度。  
- **读用快照**：同上 COW，少分配、少拷贝，缓存友好。

### 5.2 嵌入：为什么“整批一起做”

- 对当前这一批文档：先全部规范化和分块得到 `all_chunks`，然后**只调一次** `embed(all_chunk_texts)`，在嵌入器内部再按 32 条一批跑模型；最后把向量填回每个 chunk 再写入存储。  
- 调用次数从“文档数”变成“批次数”，GPU/CPU 利用率高，吞吐量上去。

### 5.3 正则和分词：为什么“只编译一次”

- 在 Chunker / KeywordIndex **初始化**时用 `re.compile(...)` 编译好，存成成员变量；之后每次分句、分词都用这个对象，不再重复编译，热路径上少浪费。

### 5.4 队列为什么用 `queue.Queue`

- 使用标准库 `queue.Queue(maxsize=N)`：`put(doc, block=False)` 写；满就接住返回 False；worker 用 `get(timeout=0.1)` 先等一条，再用 `get_nowait()` 凑满一批。行为正确、语义清晰。

## 六、循环与低效写法优化（要点）

| 原则 | 说明 |
|------|------|
| 少用循环内 append | 能写成列表推导或单次 `extend` 的，优先用推导/`extend`。 |
| 能用 NumPy 就一次算完 | 如多 query 的 argsort 用 `np.argsort(-scores, axis=1)[:, :k]` 一次得到所有 top-k。 |
| 避免临时大列表 | 不构造 `A + B` 再遍历，用两遍遍历分别更新同一结构（如 RRF 合并时的 `chunk_by_id` 与 `scores`）。 |
| 重复计算提到循环外 | 如 doc_lens、meta_base、hash 列表在循环外算一次。 |
| 批量操作替代逐项 | 删除、过滤用 set 差集或一次列表推导完成。 |

**按文件举例**（详见原 `rag/LOOP_OPTIMIZATION.md` 若需逐行对照）：

- **vector_store**：`add()` 用一次列表推导得到合法 chunk+向量，再 `extend`；`search_batch()` 用 `argsort(axis=1)` 一次覆盖所有 query。  
- **keyword_index**：IDF 用字典推导；search 预计算 `doc_lens`，内层 O(1) 查；`remove_chunk_ids` 用 set 批量删除。  
- **retriever**：元数据过滤用列表推导+谓词函数；RRF 合并两遍写同一 dict，不拼 `vector_results + keyword_results`。  
- **chunker**：最终 chunk 列表用一次列表推导 + 公共 `meta_base`。  
- **processing pipeline**：`chain.from_iterable` 展开，一次 `list(...)` 得到整批 chunk。  
- **pipeline**：去重先算 hash 列表，再统一 evict，避免每加一个就 while 一次。

## 七、极限优化：热路径、突发与精度

### 7.1 数值热路径：`rag/core.py`

- 只做两件事：**单 query 的 dot + top-k**、**多 query 的 dot + 每行 top-k**。  
- 有 Numba 时：`@njit(cache=True, fastmath=True)` 编译为机器码。无 Numba 时：用 NumPy 的 `dot` 和 `argsort`，接口不变。  
- **可替换为 Rust/C++ 的边界**：接口保持为 `(mat, q, k) -> (idx, scores)` 与 `(mat, q_mat, k) -> (idx_topk, scores_k)`；用 PyO3 或 pybind11 实现同样签名即可，业务层无感知。

### 7.2 突发保护

- 配置项 **`max_process_batch`**：单次从队列取出的文档数上限。Worker 取批时用 `min(process_batch_size, max_process_batch)`，避免单批过大导致延迟尖刺和 OOM。

### 7.3 精度与数值稳定性

- **float32**：全链路向量与分数用 float32，省内存、与常见嵌入模型一致；对相似度排序精度足够。  
- **归一化**：检索前 L2 归一化，余弦相似度用一次 dot 完成。  
- 排序 key：`operator.itemgetter(1)` 替代 lambda，在热路径上少一次 Python 调用。

## 八、小结表（一页看清）

| 目标 | 我们做了什么 | 对你/业务的好处 |
|------|--------------|-----------------|
| 和量化框架融合 | RagContextProvider + get_rag_context(..., as_of_date=current_date) | 回测时间一致、数据层统一。 |
| 写入要“实时” | 有界队列 + 非阻塞 put + 后台 daemon worker 聚批处理 | ingest_put 马上返回。 |
| 检索要稳定 | COW 更新，读只读快照、临界区极短 | 检索不因正在写入而明显变慢。 |
| 向量存和查要快、省 | 单块 numpy、预分配、写入时归一化、读用快照 | 少分配少拷贝，延迟和内存都更稳。 |
| 嵌入要高效 | 整批文档 → 整批 chunk → 一次批量 embed | 调用次数少、批大，利用率高。 |
| 热路径少浪费 | 正则编译一次、复用；core 层 Numba/NumPy | 分句/分词与 dot+top-k 高效。 |
| 突发与限流 | max_process_batch、max_vectors FIFO | 不 OOM、延迟更平滑。 |

## 九、配置与可观测性

| 能力 | 配置或接口 | 说明 |
|------|------------|------|
| 配置校验 | `RAGConfig` 在 `__post_init__` 中 `_validate()` | 启动时发现非法值，报错清晰。 |
| 内存上限 | `max_vectors` | 超过后 FIFO 淘汰，并与关键词索引同步移除。 |
| 文档去重 | `skip_duplicate_docs` + `dedup_cache_max` | 按内容 hash 跳过重复文档；LRU 缓存。 |
| 短块过滤 | `min_chunk_length` | 分块后过短片段丢弃。 |
| 查询嵌入缓存 | `query_embedding_cache_size` | 相同 query 复用嵌入（LRU）。 |
| 健康检查 | `pipeline.health_check()` | 队列长度/上限、向量条数、关键词条数、worker 是否启动。 |

## 十、未来方向与框架极限

**可后续迭代的方向**：近似最近邻 (ANN)、向量持久化、按时间过滤（created_at <= as_of_date）、文档级去重增强、查询嵌入缓存、多 worker/进程、分片/分布式、重排模型优化。

**当前客观边界**（大致）：  
- 向量条数受单机内存限制（全量点积）；再大需 ANN 或持久化/分片。  
- 检索延迟与向量条数近似线性。  
- 写入吞吐受单线程嵌入与单次批大小限制。  
- 实时可见性为“worker 处理完这一批才可被检索”，通常秒级内。  
- 时间一致性：接口有 as_of_date，检索层可按需实现时间过滤。  
- 单进程、无持久化；高可用需外加部署与持久化。

以上便于评估是否够用、何时要换方案或升级架构。
