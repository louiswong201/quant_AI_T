# 如何得到最稳健的回测结果

本文档给出**在现有框架下**可落地的做法，让回测结果更稳健、更可信，减少过拟合和「回测好看、实盘崩」的落差。

---

## 一、稳健回测清单（建议逐项做到）

| 序号 | 做法 | 说明 |
|------|------|------|
| 1 | **成本设保守** | 手续费 + 滑点略高于实盘预期（如佣金 0.0015、滑点 5–10 bps），避免回测过于乐观。 |
| 2 | **多区间验证** | 同一策略在多个**不重叠**时间段各跑一次（如 2018–2019、2020–2021、2022–2023），看收益/回撤的**分布**，不只看一段。 |
| 3 | **多标的验证** | 在多个标的或不同板块上跑同一策略，看是否多数能盈利或至少回撤可控。 |
| 4 | **参数扫描 + 分布** | 对关键参数（如均线周期）取多组取值分别回测，汇报**分位数**（如 25%/50%/75% 收益）或**最差区间**，避免只报「最优一组」。 |
| 5 | **信号与执行对齐实盘** | 设计为「T 日收盘得到信号 → T+1 执行」，回测里用下一 bar 成交，与实盘可执行性一致。 |
| 6 | **记录与复现** | 每次回测保存：数据区间、标的、策略参数、手续费/滑点配置，便于复现与审阅。 |

框架**已支持**：保守成本（`BacktestConfig` 手续费/滑点）、多标的、限价/止损；**未内置**多区间/多参数批量跑与汇总，需你在外层脚本中循环调用 `BacktestEngine.run` 并自己汇总（见下文示例）。

## 二、推荐做法详解

### 2.1 保守成本（立即可用）

```python
from quant_framework.backtest import BacktestEngine, BacktestConfig

# 方式一：使用框架预设（推荐）
config = BacktestConfig.conservative()  # 佣金 0.15%，滑点 5 bps
engine = BacktestEngine(data_manager, config=config)

# 方式二：自定义
config = BacktestConfig(
    commission_pct_buy=0.0015,
    commission_pct_sell=0.0015,
    slippage_bps_buy=5.0,
    slippage_bps_sell=5.0,
)
engine = BacktestEngine(data_manager, config=config)
```

实盘若佣金约 0.1%、滑点约 3 bps，回测用略高的值，结果更保守、更稳。

### 2.2 多区间回测（减少对单一时段的依赖）

把总区间拆成多段**不重叠**（或少量重叠）的窗口，每段单独回测，例如：

- 2018-01-01 ~ 2019-12-31  
- 2020-01-01 ~ 2021-12-31  
- 2022-01-01 ~ 2023-12-31  

对每段得到「期末净值 / 期初净值」或年化收益，再看**最小值、中位数、最大值**。若多数区间为正且最差区间可接受，比「只跑 2020–2023 且只报这一段」稳健得多。

### 2.3 多参数扫描（避免只报告「最优参数」）

对同一策略多组参数各跑一遍（可再结合多区间），例如均线 (5,20)、(10,30)、(20,60)，得到多组收益曲线或期末收益，汇报：

- 各区间/各参数下的收益或夏普；
- 按区间：min/median/max；
- 按参数：最好/最差参数出现在哪些区间。

这样能看出策略是「多数参数多数区间都还行」还是「只有某组参数在某一时段特别好」（后者过拟合风险高）。

### 2.4 多标的验证

用 `BacktestEngine.run(strategy, symbols=[...], start_date, end_date)` 对多只股票或不同板块跑同一策略，看：

- 正收益标的占比；
- 回撤分布。

若只有少数标的赚钱、多数亏损，策略稳健性存疑。

### 2.5 信号与执行时点（策略设计）

- 若策略用「当日 close」算信号并在回测里用当日 close 成交，实盘无法在收盘前拿到 close，会高估回测。
- 建议：**T 日收盘后得到信号，T+1 日开盘或市价执行**；回测时信号在 T 日 bar 产生，成交在 T+1 bar（引擎已支持按下一 bar 的 OHLC 成交）。

## 三、示例：多区间 + 多参数汇总（思路与用法）

下面用「多区间 + 多组均线参数」做批量回测并汇总，得到**更稳健的结论**（不只看一条曲线、一组参数）。

- 时间窗口：例如 3 段不重叠区间。  
- 参数：例如 `(short, long) = (5,20), (10,30), (20,60)`。  
- 对每个 (区间, 参数) 调用一次 `engine.run(...)`，记录 `final_value`、`initial_capital`，算区间收益率。  
- 汇总：按区间看「3 组参数下的 min/median/max 收益」；按参数看「3 个区间下的 min/median/max 收益」。

这样可以看到：  
- 若某参数在 3 个区间里 2 个亏损 → 该参数不稳；  
- 若某区间 3 组参数都亏 → 该时段策略整体失效，需警惕。

**框架内置**：`run_robust_backtest(engine, strategy_factory, symbol, windows, param_sets)` 可直接调用，返回每 (窗口, 参数) 的收益与 by_param / by_window 汇总。可运行示例见 **examples/example_robust_backtest.py**（需本机有数据与依赖）。

## 四、小结

| 目标 | 方法 |
|------|------|
| 结果更稳、更可信 | 保守成本 + 多区间 + 多标的 + 多参数，看**分布**不只看单点。 |
| 减少过拟合 | 不选「历史最优参数」做唯一结论；汇报分位数与最差情况。 |
| 与实盘更接近 | 信号/执行时点对齐（T 信号 → T+1 执行）；成本略高估。 |

框架提供**单次回测**的正确性与可配置成本；**「最稳健的回测结果」**来自你在其之上的**多区间、多标的、多参数验证与保守报告**。更多回测与实盘差异说明见 [BACKTEST_VS_LIVE.md](BACKTEST_VS_LIVE.md)。

### 回测与实盘/纸交易一致性清单（减少与 Freqtrade 类似的落差）

- **数据**：每 bar 只使用「到当前 bar 为止」的切片（本框架回测已保证）；实盘驱动时传入的 `data` 必须同样是「到当前时刻为止」，且**历史 K 线数 ≥ 策略最大 lookback**（如 MA60 则至少 60 根，建议略多）。  
- **指标**：若实盘按 bar 增量算指标，公式须与回测一致；数据源与回测尽量同口径。  
- **成本**：回测使用保守手续费与滑点（≥ 实盘预期）。  
- **执行**：回测已用「下一 bar 成交」；实盘按「T 日信号 → T+1 执行」与之对齐；多标的时实盘处理标的/信号的顺序建议与回测一致。  

详见 [FREQTRADE_BACKTEST_LIVE_ANALYSIS.md](FREQTRADE_BACKTEST_LIVE_ANALYSIS.md)。
