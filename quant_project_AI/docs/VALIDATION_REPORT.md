## 三、回测逻辑正确性校验

### 3.1 发现并修复的 Bug

#### Bug 1: 手续费未从 cash 中实际扣除

| 项目 | 详情 |
|------|------|
| 现象 | `strategy.buy()` 执行 `cash -= price * shares`，但 commission 仅记录在 trades 表，从未从 cash 扣除 |
| 影响 | 回测收益虚高。以 AAPL MA(5,20) 为例，15 笔交易累计 commission $19,029 完全未计入 |
| 涉及代码 | `backtest_engine.py` 中所有 4 条交易路径（市价买/卖 + 限价买/卖） |
| 修复方式 | 在每次 `strategy.buy()`/`strategy.sell()` 之后立即执行 `strategy.cash -= commission` |

#### Bug 2: portfolio_value 使用了 pre-trade 陈旧值

| 项目 | 详情 |
|------|------|
| 现象 | `update_portfolio_value()` 在 bar 开头调用（交易前），交易完成后 cash/positions 已变化，但 PV 仍是旧值 |
| 影响 | 持仓变动 bar 的 PV 偏差最大 ~$1,900（不等于 cash + positions × close） |
| 修复方式 | 在 bar 末尾（所有交易完成后）再次调用 `update_portfolio_value(current_prices)` |

## 六、性能瓶颈深度分析

### 6.1 框架原生引擎 vs Numba 极速扫描

| 引擎 | 单次回测耗时 | 7.9M 回测预估耗时 | 吞吐量 |
|------|-----------|----------------|--------|
| **Numba 极速扫描** | **0.002 ms** | **16 秒** | **493,915/秒** |
| 框架原生 BacktestEngine | 133.7 ms | ~294 小时 | ~7.5/秒 |
| **加速倍数** | | | **66,134×** |

### 6.2 框架原生引擎慢在哪？

以单次 MA 回测 (752 bars) 的 133.7ms 为例，时间分解：

| 开销项 | 估计耗时 | 占比 | 说明 |
|--------|---------|------|------|
| DataFrame iloc 切片 | ~50ms | 37% | 每 bar 切片 `data.iloc[:i+1]`，752 次，含索引重建 |
| 指标计算 (calculate_indicators) | ~30ms | 22% | 每次 run 都重新计算全部指标 (MA/RSI/MACD/BB/ATR 等) |
| 策略对象开销 | ~15ms | 11% | 策略初始化、positions dict 操作、portfolio_value 更新 |
| 信号解析与订单状态机 | ~15ms | 11% | dict 解析、order state machine、pending order 管理 |
| 数据加载 (CSV read) | ~10ms | 7% | pd.read_csv + 类型转换 |
| 流动性/冲击模型 | ~8ms | 6% | max_participation_rate 检查、market impact 计算 |
| 其他 (manifest, GC 等) | ~6ms | 4% | build_run_manifest、Python GC |

### 6.3 Numba 扫描为什么快？

| 优化策略 | 效果 |
|---------|------|
| **指标一次性预计算** | 199 个 MA/EMA/RSI 全部提前算好存入 2D ndarray，回测时 O(1) 查表 |
| **Numba JIT 编译** | 内层 752-bar 循环编译为原生机器码，无 Python 解释器开销 |
| **纯标量运算** | 无 DataFrame、无 dict、无对象分配，全部 float64 标量 |
| **无 GC 压力** | 整个扫描过程几乎无 Python 对象创建 |
| **缓存友好** | 连续内存访问 (contiguous ndarray)，CPU cache 命中率高 |

### 6.4 MACD 扫描为什么占 98%？

| 因素 | 详情 |
|------|------|
| 参数空间巨大 | 2,607,099 组合/标的 = MA 的 132 倍 |
| Signal EMA 不可预计算 | signal 线依赖 (fast, slow) 对生成的 MACD line，必须每个 combo 重新算 |
| Python for 循环 | 260 万次函数调用的 Python 解释器开销累积 |

### 6.5 进一步加速方向

| 方案 | 预期加速 | 复杂度 |
|------|---------|--------|
| 预计算 MACD line (fast×slow) 存入 2D 数组 | 2-3× | 低 |
| multiprocessing 3 标的并行 | 3× | 低 |
| Numba 批量 signal EMA (向量化广播) | 5-10× | 中 |
| Cython/Rust 替代 Python 外层循环 | 10-20× | 高 |
| GPU (cupy/CUDA) 批量并行 | 50-100× | 高 |
